# 联机实机回归清单（M4/M5）

## 0. 准备
1. 使用两个浏览器（或设备）模拟至少 2 个真人账号。
2. 确认 `src/firebase-config.js` 已填写，匿名登录可用。
3. 至少保留 1 个房主窗口常驻，用于驱动 host loop。
4. 建议先执行：
   - `npm.cmd test`
   - `npm.cmd run test:release`
   - `npm.cmd run test:stability`
5. 可先运行 `npm.cmd run smoke:prepare` 生成当日记录文件：
   - 默认路径：`漳麻联机版/docs/smoke-logs/YYYY-MM-DD.md`

## 1. 建房入房
1. A 创建房间，记录房间码。
2. B 输入房间码加入。
3. 检查 4 座位是否自动补 AI，不足真人时可直接开局。
4. 检查房间页牌桌是否显示四家牌河/副露/花牌。
5. 检查“同步诊断”的 `meta.version / game.version` 是否单调递增，无回退告警。

## 2. 掉线接管与重连夺回
1. 让 B 入座后关闭 B 页面。
2. A 观察 B 座位是否变为 `AI接管`，牌局是否继续推进。
3. B 使用同一浏览器会话重新进入房间。
4. 检查 B 是否自动夺回原座位，控制权从 `bot` 切回 `human`。

## 3. 房主迁移
1. A 作为当前房主，关闭 A 页面。
2. 观察在线真人座位中是否按座位顺序接任 `hostUid`。
3. 新房主窗口继续推进对局，确认 `game.version` 持续递增。
4. 检查“同步诊断”事件流出现 `房主迁移` 记录。

## 4. 关键规则链路
1. 抢杠胡：补杠/暗杠触发抢杠窗口，验证“有人胡”和“无人胡”两条分支。
2. 三金倒强制胡：出现三金倒时仅允许胡，不可通过弃牌绕过。
3. 吹风：庄家前两手目标牌，三家全跟触发即时分结算。
4. 游金/三金倒叠加：补杠或补花补牌命中时可叠加“杠上开花/花开富贵”。

## 5. 调试面板权限
1. 非房主检查：调试开关不可用，不能发送调试动作。
2. 房主开启调试后：可发送模板动作，并可指定座位发送动作。
3. 关闭调试后：面板恢复禁用。

## 6. Firebase 规则验收
1. 非成员 uid 直接读取房间路径应被拒绝。
2. 非房主尝试写 `meta/game` 应被拒绝。
3. 成员提交 `actions/*` pending 成功；非成员提交失败。
4. `presence/{uid}` 仅本人可写，跨 uid 写入失败。

## 7. 接下来要做（按顺序执行）
1. 先执行第 1-3 节，确认“可开局、可迁移、可重连夺回”主链路稳定。
2. 再执行第 4 节四条规则链路，至少每条成功 1 次。
3. 执行第 6 节规则验收，记录每条“应拒绝/应放行”结果。
4. 执行第 5 节权限验收，确认非房主无法滥用调试入口。
5. 最后做 GitHub Pages 冒烟：大厅 -> 建房/入房 -> 开局 -> 一局结束 -> 下一局。

## 7.1 核心问题对齐步骤（必须同步）
1. 执行完第 7 节后，更新根目录 `CORE_STATUS.md` 的两个核心结论。
2. 若第 1-4 节任一失败，`CORE_STATUS.md` 必须标记“未达标 + 阻塞项”。
3. 仅当第 1-6 节全部通过，才允许把“联机与单机功能一致”标记为“可发布级”。
